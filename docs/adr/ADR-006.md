# ADR-006: Multi-Institution Parser Registry

**Status:** Accepted  
**Date:** June 8, 2026  
**Context:** Phase 2 blocked pending CashApp layout inspection

---

## Context

The system initially assumed Chime PDFs only. After Phase 1 completion, inspection of CashApp PDFs revealed a **fundamentally different layout** that cannot be parsed with the same regex patterns.

### Chime Format (6 columns)
```
1/31/2025 Transfer to Savings Account Transfer $200.00 $500.00 1/31/2025
DATE      DESCRIPTION                 TYPE     AMOUNT   BALANCE SETTLEMENT_DATE
```

### CashApp Format (5 columns)
```
Apr 8 Wal Mart West Palm Bea FL Cash App Card $0.00 $0.83
DATE  DESCRIPTION                           DETAILS FEE   AMOUNT
```

**Key differences:**
- Date format: `M/D/YYYY` vs `Mon D`
- Amount polarity: negative sign vs `+`/`$` suffixes
- Fee column present in CashApp, absent in Chime
- Different transaction type vocabularies

## Decision

Implement a **parser registry pattern** with institution-specific parsers.

### Architecture

```
parse_pdf(path)  # entry point
    ↓
[_detect_institution(path)]  # chime | cashapp | unknown
    ↓
[InstitutionParser.parse(path)]  # registry dispatch
    ↓
[Institution-specific regex/text extraction]
    ↓
ChimeTransaction (normalized output)
```

### Implementation

1. **Base class** `BaseParser` — defines `parse(path: Path) -> list[ChimeTransaction]` interface
2. **Registry** `PARSER_REGISTRY: dict[str, BaseParser]` — maps institution name to parser instance
3. **Detectors** `INSTITUTION_DETECTORS: list[tuple[Pattern, str]]` — regex patterns to identify institution from filename/content
4. **Concrete parsers**:
   - `ChimeParser` — existing regex logic from Phase 1
   - `CashAppParser` — new parser for CashApp layout

### Data Model Update

Add `source_institution` to `ChimeTransaction` and `transactions` table:

```python
@dataclass(frozen=True, slots=True)
class ChimeTransaction:
    # ... existing fields ...
    source_institution: str  # "chime" | "cashapp"
```

```sql
ALTER TABLE transactions ADD COLUMN source_institution TEXT NOT NULL DEFAULT 'chime';
```

### Registry Registration

```python
# parsers/__init__.py
from chime_ingestor.parsers.base import BaseParser
from chime_ingestor.parsers.chime_parser import ChimeParser
from chime_ingestor.parsers.cashapp_parser import CashAppParser

PARSER_REGISTRY: dict[str, BaseParser] = {
    "chime": ChimeParser(),
    "cashapp": CashAppParser(),
}

INSTITUTION_DETECTORS = [
    (re.compile(r"chime", re.I), "chime"),
    (re.compile(r"cashapp|cash.app|square", re.I), "cashapp"),
]

def detect_institution(path: Path) -> str:
    filename = path.name.lower()
    for pattern, institution in INSTITUTION_DETECTORS:
        if pattern.search(filename):
            return institution
    return "unknown"

def get_parser(institution: str) -> BaseParser:
    if institution not in PARSER_REGISTRY:
        raise ValueError(f"No parser for institution: {institution}")
    return PARSER_REGISTRY[institution]
```

## Consequences

**Positive:**
- Clean separation of institution-specific parsing logic
- Easy to add new institutions (Fidelity, Schwab, etc.) — just implement `BaseParser`
- Tests isolated per institution
- `source_institution` enables institution-aware queries (e.g., "show only CashApp")

**Negative:**
- Additional file structure: `parsers/` directory with base + concrete implementations
- Refactor required: move existing Chime logic to `ChimeParser` class
- More tests needed: each parser needs its own test file

**Migration:**
- Existing `parser.py` logic becomes `parsers/chime_parser.py`
- `parse_pdf()` becomes a thin dispatch wrapper
- Existing tests move to `tests/test_chime_parser.py`
- New `tests/test_cashapp_parser.py` for CashApp coverage

## Blockers Removed

Phase 2 can now proceed with:
1. ✅ Multi-institution parser architecture defined
2. ✅ `source_institution` schema change specified
3. ✅ CashApp layout inspected and documented

## Deferred

- CashApp parser implementation — Phase 2.5 (after SQLite batch ingestor works for Chime)
- Additional institutions — future phases as needed
